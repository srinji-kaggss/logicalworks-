# Good-Code Acquisition Pipeline — Grounding

Research for a pipeline to ACQUIRE → FILTER → VECTORIZE → JSON-structure → LOG a corpus
of high-quality internet code, to train a code-evaluator ML.

Method: web-grounded (WebSearch + WebFetch on official sources). Firecrawl search/scrape and
`crwl`/Bash were unavailable this session (402 quota / sandbox denial); sources are cited inline.
Date: 2026-06-01.

## 2. "Good code" quality signals → data source

| Signal | What it proxies | Extract from |
|---|---|---|
| **Stars / forks** | community trust, popularity | GitHub REST/GraphQL API (`stargazers_count`, `forks_count`); GH Archive `WatchEvent`/`ForkEvent`; The Stack v2 keeps star/fork counts as dedup tiebreaker |
| **CI-green status** | builds + passes tests | GitHub Checks API / Actions API (`conclusion=success`); GH Archive `CheckRunEvent`/`StatusEvent`; presence of `.github/workflows` |
| **Test presence / coverage** | verified behavior | repo file tree (test dirs, `*_test.*`, `pytest`/`jest` configs); coverage badges/reports; codecov API |
| **Maintenance recency** | not abandoned | GitHub API `pushed_at`/`updated_at`; GH Archive `PushEvent` recency; latest commit date (Stack v2 tiebreaker) |
| **License cleanliness** | legally trainable | ScanCode / Blue Oak classification; SPDX in LICENSE file; GitHub `license` field; The Stack restricts to permissive |
| **Review density** | peer scrutiny | GitHub API PR review count; GH Archive `PullRequestReviewEvent` / `PullRequestReviewCommentEvent` per merged PR |
| **Security-audit signals** | hardened, no known CVEs | GitHub Dependabot/code-scanning alerts API; OpenSSF Scorecard; secret-scanning; absence of flagged CVEs |
| **Issue resolution** | responsiveness/health | GitHub API closed/open issue ratio + time-to-close; GH Archive `IssuesEvent`/`IssueCommentEvent` |
| **Heuristic file quality** | well-formed source | computed on the blob: line length (drop >1000-char lines / mean line len), alphanumeric ratio, fraction of comments, autogenerated/minified detection, XML/HTML-in-code filter (per StarCoder/StarCoder2 pipeline) |
| **Contributor count / bus factor** | sustained collaboration | GitHub API contributors; GH Archive distinct `actor` on `PushEvent` |

GH Archive is the time-series of GitHub events (15+ types: push, fork, watch/star, issue, PR,
review, comment, member, etc.), available as hourly JSON and a BigQuery public dataset updated
hourly — the cheapest way to compute review density, recency, and issue-resolution at scale
without hammering the REST API rate limit. (gharchive.org)

## 3. Dedup + decontamination

- **Exact dedup**: hash the normalized blob (whitespace-insensitive) and drop exact matches —
  CodeParrot github-code does this; cheap first pass. (HF `codeparrot/github-code`)
- **Near-dup via MinHash + LSH** (the standard at scale):
  - *StarCoder / The Stack-dedup*: MinHash with **256 permutations**, LSH to cluster, then exact
    Jaccard inside clusters with threshold **0.85**; initial pipeline used **5-grams** and a 0.7
    threshold. (arxiv 2305.06161)
  - *The Stack v2 / StarCoder2*: 5-grams, Jaccard threshold **0.7**; tiebreaker keeps the file from
    the repo with higher **stars/forks** or latest commit date — i.e. dedup preserves the
    higher-quality copy. (arxiv 2402.19173)
- **Benchmark decontamination** (avoid test-set leakage into an evaluator): remove files
  containing docstrings/solutions from **HumanEval** and **MBPP**, docstrings from **APPS**,
  questions from **GSM8K**, prompts from **DS-1000**. Match on substring/normalized n-gram of the
  held-out eval items. (arxiv 2305.06161)
- **PII redaction**: StarPII model used to redact names/emails/keys/IPs before training.
  (arxiv 2305.06161)
- **Malware removal**: StarCoder2 pipeline removes malicious code. (arxiv 2402.19173)
- Scaling note: trillion-scale dedup is the dominant cost; memory-efficient variants
  (e.g. LSHBloom, arxiv 2411.04257) trade a controlled false-positive rate for far less RAM.

## 4. Code vectorization / embedding

Token-level vs structure-aware:
- **Token-level** (treat code as token sequence): CodeBERT, StarEncoder, and most retrieval
  embedders. Simple, fast, language-agnostic. (unite.ai code-embedding guide)
- **Structure-aware**: **GraphCodeBERT** — first pretrained model to use code structure, but uses
  **data-flow graph** ("where-the-value-comes-from") rather than full AST, because AST adds a deep
  hierarchy and cost; graph-guided masked attention injects structure into the Transformer. CFG/DFG
  graph methods and StructCoder (AST-aware decoder) are the heavier alternatives. (arxiv 2009.08366;
  arxiv 2206.05239) For an *evaluator*, data-flow/structure signals are worth it because quality
  correlates with structural soundness, not just surface tokens.

Embedding model options:

| Model | Dim | Context | Weights | Local feasibility |
|---|---|---|---|---|
| **CodeBERT** | 768 | 512 | open (125M) | trivial on-device (CPU/small GPU) |
| **StarEncoder** (BigCode) | 768 | 1024 | open (~125M) | trivial on-device; reported slightly > CodeBERT on embedding comparison |
| **jina-embeddings-v2-base-code** | 768 | 8192 | open | on-device; long-context for whole files |
| **jina-code-embeddings (1.5B)** | (Matryoshka) | long | open | mid GPU / quantized on-device |
| **nomic-embed-code (7B)** | up to 768, Matryoshka down to 64 | long | open | needs GPU; quantize for on-device |
| **CodeSage-large** | — | 1K | open | on-device |
| **voyage-code-3** | 2048/1024/512/256 | **32K** | **API only** | not on-device; best retrieval scores (beats OpenAI-v3-large +13.8%, CodeSage-large +16.8% over 32 code datasets); int8/uint8/binary quantization |
| **OpenAI text-embedding-3-large** | up to 3072 | 8K | API only | not on-device |

On-device pick for Jarvis (24GB RAM, MLX): an open token-level model with Matryoshka
(nomic / jina-code) lets you store 256–768-dim vectors cheaply; quantize to int8/binary for the
index. Reserve voyage-code-3 for an offline high-accuracy re-rank if API use is acceptable.
(Voyage AI blog 2024-12-04; modal.com 6-best-code-embeddings; pecollective specs 2026)

## 5. Legal / license constraints

- **Permissive (MIT/BSD/Apache-2/ISC/CC0/Unlicense)** is the safe training base; The Stack and
  CommitPack restrict to these via ScanCode/Blue Oak classification. Apache-2/BSD/MIT still require
  **attribution + license-notice retention** if you redistribute substantial verbatim code.
- **Copyleft (GPL/AGPL/LGPL/MPL/EPL)** is the contested zone. CodeParrot *includes* GPL; The Stack
  *excludes* it. Training on copyleft and emitting near-verbatim copies risks license-propagation
  claims. Safer: exclude strong copyleft from any corpus a generative/evaluator model could
  reproduce.
- **Provenance debate / Copilot litigation**: *Doe v. GitHub* (filed 2022) alleges training on
  GitHub code without honoring MIT/GPL attribution + DMCA §1202 CMI-stripping. By 2024 most IP/DMCA
  claims were dismissed; **breach-of-license / breach-of-contract claims survived**, signaling
  courts treat OSS licenses as enforceable contracts. Ninth Circuit accepted an appeal Dec 2024 on
  whether DMCA §1202(b) reaches non-identical AI outputs; district case stayed.
  (saverilawfirm.com; theregister.com 2024-01-12; en.wikipedia.org/GitHub_Copilot)
- **EU AI Act**: GPAI providers must publish a summary of training-data categories and comply with
  EU copyright; transparency obligations bite 2025–2026.
- Practical guidance: (1) permissive-only by default, copyleft opt-in with isolation; (2) keep an
  immutable **provenance record per file** (repo, commit SHA, license, SPDX, retrieval timestamp);
  (3) honor an **opt-out** path (reuse BigCode "Am I in The Stack" list); (4) retain license
  notices; (5) never store secrets/PII — redact at ingest.

## 6. Recommended staged pipeline

Filter criteria = the §2 signal taxonomy.

1. **ACQUIRE** — Base corpus from **The Stack v2-dedup** (permissive, provenance, opt-out built in).
   Enrich/extend via **GH Archive on BigQuery** for repo-level signals (stars, forks, PR reviews,
   issue resolution, recency). Optionally pull live deltas via GitHub GraphQL for fresh repos.
   Record per file: repo, commit SHA, path, license/SPDX, source snapshot, retrieved-at.
2. **QUALITY-FILTER** — Two tiers:
   - *Repo gate*: stars/forks ≥ threshold, CI-green, tests present, `pushed_at` recent,
     permissive license, low open/closed-issue staleness, review density > 0, no open critical
     security alerts (OpenSSF Scorecard / Dependabot).
   - *File heuristics* (StarCoder2): drop >1000-char lines, low alphanumeric ratio, autogenerated/
     minified, XML/HTML-in-code; keep reasonable comment fraction. Emit a per-file quality vector
     (the labels your evaluator learns from).
3. **DEDUP + DECONTAMINATE** — exact-hash pass → MinHash(256 perm)+LSH near-dedup, Jaccard 0.85,
   5-grams; tiebreak keep higher-star/newer copy. Decontaminate against HumanEval/MBPP/APPS/
   DS-1000/GSM8K. Redact PII (StarPII-style). Drop flagged malware.
4. **VECTORIZE** — embed surviving files with an open code embedder (token-level nomic/jina-code,
   Matryoshka 256–768; add GraphCodeBERT data-flow features if structure-quality matters). Store
   int8/binary-quantized vectors for the on-device index.
5. **JSON-STRUCTURE** — one record per file:
   `{id, repo, commit_sha, path, lang, license/spdx, source_snapshot, retrieved_at, quality_signals{stars,forks,ci_green,tests,recency,review_density,issue_resolution,security_score,heuristics{...}}, embedding_ref, dedup_cluster_id, decontam_flag, pii_redacted}`.
6. **LOG (hash-chained)** — append-only audit log; each entry hashes the record + previous entry
   hash (Merkle/blockchain-style chain) so the corpus and every filter decision are tamper-evident
   and reproducible — matches the content-addressed E2EE fact-log substrate. Persist opt-out
   exclusion list and license provenance here for compliance.
